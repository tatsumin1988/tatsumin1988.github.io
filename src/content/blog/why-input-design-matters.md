## UI改善だけでは定着しない

現場改善の相談で多いのは、「入力画面をもっと使いやすくしたい」という要望です。もちろん見た目や操作性は大事ですが、それだけでは改善は長続きしません。理由は単純で、入力の責任が曖昧なままだと、どんなに UI を整えても「誰も最後まで責任を持たない情報」が増えるからです。

> 現場の入力負荷は、項目数の多さよりも「なぜ自分が今これを書くのか」が曖昧なときに急に重くなります。

現場で起きがちな失敗は次のようなものです。

- いつ書くべきかが曖昧で、後追い入力になる
- 誰が確認して完了にするのか決まっていない
- 後工程がその情報をどのように使うか見えない

この状態では、入力画面をいくら美しくしても「入力が面倒」という感覚は残ります。

## 先に決めるべき3つの観点

### 誰が一次入力者か

一次入力者を曖昧にすると、「気づいた人が書く」「あとでまとめて書く」が発生します。これは記録の粒度を壊し、後から分析できない原因になります。

私がまず確認するのは、次の3点です。

- そのイベントを最初に観測する人は誰か
- その人が現場で入力できる時間はどれくらいか
- その人が入力できない場合の代替ルートはあるか

### どこまでを必須にするか

入力項目を増やせば分析しやすくなるように見えますが、実際には空欄が増えて精度が下がります。必須にすべきなのは、後工程の判断に直結する項目だけです。

| 項目種別 | 必須にする基準 | 例 |
| --- | --- | --- |
| 発生事実 | 後から変えられない | 時刻、設備、異常区分 |
| 判断補足 | 後工程で確認可能 | 所感、備考 |
| 分析用情報 | 後で補完できる | 詳細分類、タグ |

### 誰が確認して締めるか

記録は入力された瞬間に完成しません。確認者が不在だと、入力ミスが放置され、現場は「書いても信用されないデータ」を持つことになります。完了条件を定義しない記録設計は、改善運用を途中で止めます。

## 実装時の考え方

画面設計に入る前に、私は入力イベントを次のような形で整理します。

```ts
export type ShiftEvent = {
  occurredAt: string;
  machineId: string;
  eventType: "stop" | "quality" | "handoff";
  ownerId: string;
  note?: string;
  confirmedBy?: string;
};
```

ここで重要なのは、見た目ではなく「入力責任」と「確認責任」が型として分かれていることです。画面が変わっても、この構造が崩れなければ運用は保ちやすくなります。

## 現場ヒアリングで必ず確認すること

- 記録が無いと誰が困るのか
- 記録が遅れると何が判断できなくなるのか
- 記録が間違っていた場合、誰が修正可能なのか

この3つが曖昧なまま UI を作ると、入力画面は増えても改善は進みません。

## まとめ

UI は改善の最後に効いてくる要素です。先に必要なのは、誰が、どのタイミングで、どこまで持つべき情報かを定義することです。入力責任が整理されると、項目設計、画面設計、確認フローが一気につながります。
